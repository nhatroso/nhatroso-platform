### 0) MVP scope mapping (SRS ‚Üí SDD)

**In-scope MVP epics (Must):**

- Account: registration, login/logout, RBAC (Owner vs Tenant)
- Owner: property/room management, pricing over time, contracts
- OCR: capture photo, OCR numeric + confidence, validate vs previous, manual correction/override
- Billing & invoices: generate invoices, status workflow, tenant invoice view
- Platform/Security: audit trail, password hashing + JWT, tenant data privacy

---

### S1. Detailed design principles

- **Single source of truth:** Pricing, readings, contracts drive invoices.
- **Append-only audit:** Critical events are always logged.
- **Deterministic billing:** No overlapping pricing effective periods.
- **Role isolation:** Tenant can only see own invoices. Owner can only see owned buildings/rooms/contracts.
- **Idempotency:** Invoice generation and webhooks must be safe to retry.

### S3. Data model (PostgreSQL) ‚Äî MVP tables

> Names are suggestions. Keep fields minimal but sufficient for requirements.

#### S3.1 Users, auth, and roles

- `users`
  - `id (i32 serial, pk)` ‚Üê integer ID, not UUID
  - `name (text, nullable)`
  - `email (text, unique, nullable)`
  - `phone (text, unique, nullable)`
  - `password_hash (text)`
  - `role (OWNER|TENANT)`
  - `created_at`, `updated_at`

- `refresh_tokens`
  - `id (i32, pk)`
  - `user_id (fk users.id)`
  - `jti (text, unique)`
  - `expires_at (timestamptz)`
  - `revoked_at (timestamptz, nullable)`
  - `created_at`

#### S3.2 Property / room management

- `buildings` ‚Äî `id (i32)`, `owner_id`, `name`, `address`, `status (ACTIVE|ARCHIVED)`, timestamps
- `floors` ‚Äî `id (i32)`, `building_id`, `identifier`, `status (ACTIVE|ARCHIVED)`
- `rooms` ‚Äî `id (i32)`, `building_id`, `floor_id (nullable)`, `code`, `status (VACANT|DEPOSITED|OCCUPIED|MAINTENANCE|ARCHIVED)`, timestamps
  - `UNIQUE(building_id, code)`

> Note: A `blocks` table was created and subsequently dropped (migration `m20260317_141521_remove_blocks.rs`).

#### S3.3 Service catalog, room-services, and effective pricing

- `services` ‚Äî `id (i32)`, `owner_id`, `name`, `unit`, `status (ACTIVE|ARCHIVED)`, `created_at`
- `room_services` ‚Äî `id (i32)`, `room_id`, `service_id`, `price_rule_id (nullable)`, `status (ACTIVE|ARCHIVED)`, `created_at`
  - Links a service to a room; optionally references the active `price_rule`.
- `price_rules` ‚Äî `id (i32)`, `owner_id`, `room_id`, `service_id`, `unit_price (numeric)`, `effective_start (date)`, `effective_end (date, nullable)`, `is_active (bool)`, `created_at`
  - Effective-dated pricing per `(room_id, service_id)`. Ranges must not overlap.

#### S3.4 Contracts (1 active contract per room)

- `contracts` ‚Äî `id (i32)`, `owner_id`, `room_id`, `start_date`, `end_date (nullable)`, `billing_cycle (MONTHLY)`, `status (ACTIVE|ENDED)`, `rental_period (int)`, `room_code (text snapshot)`, `room_address (text snapshot)`, `created_at`
- `contract_tenants` ‚Äî `contract_id`, `tenant_id`, PK `(contract_id, tenant_id)`
  - Constraint: only 1 contract `ACTIVE` per `room_id`.

#### S3.5 Meters and readings (with evidence)

- `meters` ‚Äî `id (i32)`, `room_id`, `type (ELECTRIC|WATER)`, `serial_no (nullable)`
- `meter_readings` ‚Äî `id (i32)`, `meter_id`, `period (char(7), YYYY-MM)`, `photo_url (nullable)`, `reading_value (numeric, nullable)`, `usage (numeric, nullable)`, `status (PENDING|ACCEPTED|REJECTED)`, `created_at`
- `meter_request_configs` ‚Äî `id (i32)`, `building_id`, `deadline_day (int)`, `period_offset (int)`, `created_at`, `updated_at`
  - Configures the submission deadline per building.
- `meter_requests` ‚Äî `id (i32)`, `building_id`, `period (char(7))`, `status (PENDING|COMPLETED|CANCELLED)`, `created_at`, `updated_at`
  - Represents a monthly meter reading request cycle per building.

> Note: No `reading_overrides` table in actual implementation. No `ocr_raw_value`, `ocr_confidence` columns ‚Äî manual reading workflow only.

#### S3.6 Invoices and line items

- `invoices` ‚Äî `id (i32)`, `owner_id`, `room_id`, `contract_id`, `period (char(7))`, `status (UNPAID|PAID|VOIDED)`, `total_amount (numeric)`, `created_at`, `updated_at`
  - Constraint: `UNIQUE(contract_id, period)`
- `invoice_details` ‚Äî `id (i32)`, `invoice_id`, `label (text)`, `amount (numeric)`
  - Simple key-value line items (label + amount). No item_type enum or quantity/unit_price breakdown.
- `invoice_status_histories` ‚Äî `id (i32)`, `invoice_id`, `from_status`, `to_status`, `reason (nullable)`, `created_at`

> Note: `invoice_lines` (with `item_type`, `quantity`, `unit_price`) was a planned design but actual implementation uses `invoice_details` (label + amount). `PENDING_CONFIRMATION` status not in actual schema.

#### S3.7 Audit log (append-only)

- `audit_events`
  - `id (uuid, pk)`
  - `owner_id (fk users.id, nullable)`
  - `actor_user_id (fk users.id, nullable)`
  - `action_type (text)`
  - `target_type (text)`
  - `target_id (uuid, nullable)`
  - `ip (text, nullable)`
  - `user_agent (text, nullable)`
  - `reason (text, nullable)`
  - `created_at`

### S5. Key sequences / flows (MVP)

#### S5.1 Registration / Login / RBAC

1. Register/login returns tokens with role claim `OWNER` or `TENANT`.
2. API guard validates JWT.
3. Policy checks:
   - Owner endpoints require role Owner.
   - Tenant endpoints require role Tenant.
   - Resource checks ensure `owner_id` matches.

#### S5.2 Effective pricing (no overlaps)

1. Owner creates a new `price_rule` for (room, service) with `effective_start`.
2. Service checks latest rule.
3. If overlap would occur ‚Üí reject `409 OVERLAPPING_EFFECTIVE_PERIOD`.
4. Otherwise, set previous rule `effective_end = new_start - 1 day`.

#### S5.3 Capture ‚Üí OCR ‚Üí validate vs previous ‚Üí finalize

1. `POST /readings/capture`: store image; create reading (DRAFT); enqueue OCR job.
2. OCR service returns `{value, confidence}`.
3. API validates numeric:
   - non-numeric ‚Üí NEEDS_REVIEW
   - numeric ‚Üí RECOGNIZED
4. Compare with previous finalized reading:
   - if decreased ‚Üí NEEDS_REVIEW
   - else allow finalize
5. Owner manually corrects (if needed) and finalizes.

#### S5.4 Generate invoices (idempotent)

1. `POST /invoices/generate?period=` loads active contracts.
2. For each contract:
   - if invoice exists ‚Üí skip / return conflict
   - else create invoice
3. Add lines:
   - RENT (fixed)
   - ELECTRIC/WATER (delta from readings)
   - SERVICES (from configured price rules)
4. If missing reading or negative consumption ‚Üí set invoice `PENDING_CONFIRMATION`.
5. Else set invoice `UNPAID`.

### S6. Security, privacy, and audit (MVP)

- **Password hashing:** Argon2id.
- **Transport:** TLS 1.2+.
- **Signed URLs:** photo/document links must expire ‚â§ 15 minutes.
- **Tenant isolation:** tenant invoice endpoints verify ownership by contract_tenants.
- **Audit events:** login, reading override/correction, invoice status changes, contract create/end.
