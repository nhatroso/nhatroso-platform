### System Design Document (SDD) — MVP Only

<aside>
🏗️

This SDD is the **implementation-ready** design for **MVP Only** scope, derived from the SRS “Must” requirements.

It defines module boundaries, core data model, API contracts, key sequences, security/authorization, error handling, and test plan.

</aside>

### 0) MVP scope mapping (SRS → SDD)

**In-scope MVP epics (Must):**

- Account: registration, login/logout, RBAC (Owner vs Tenant)
- Owner: property/room management, pricing over time, contracts
- OCR: capture photo, OCR numeric + confidence, validate vs previous, manual correction/override
- Billing & invoices: generate invoices, status workflow, tenant invoice view
- Platform/Security: audit trail, password hashing + JWT, tenant data privacy

---

### S1. Detailed design principles

- **Single source of truth:** Pricing, readings, contracts drive invoices.
- **Append-only audit:** Critical events are always logged.
- **Deterministic billing:** No overlapping pricing effective periods.
- **Role isolation:** Tenant can only see own invoices. Owner can only see owned buildings/rooms/contracts.
- **Idempotency:** Invoice generation and webhooks must be safe to retry.

---

### S2. Module structure (NestJS) and responsibilities (MVP)

**Recommended monorepo**

- `apps/api` (NestJS)
    - `AuthModule` (register/login/logout/refresh)
    - `UsersModule` (profiles)
    - `RbacModule` (guards + permission mapping)
    - `PropertyModule` (buildings/floors/rooms)
    - `ContractModule` (contracts + room status updates)
    - `ServiceCatalogModule` (services)
    - `PricingModule` (effective-dated price rules)
    - `MeterModule` (meters + readings)
    - `OcrModule` (OCR job orchestration)
    - `BillingModule` (invoice generation + invoice lines + state machine)
    - `TenantBillingModule` (tenant invoice list/detail)
    - `AuditModule` (append-only audit events + export)
- `apps/ocr-service` (FastAPI)
    - `POST /ocr/jobs` (accept image URL)
    - `POST /ocr/callback` (return result to API)

### S3. Data model (PostgreSQL) — MVP tables

> Names are suggestions. Keep fields minimal but sufficient for requirements.

#### S3.1 Users, auth, and roles

- `users`
    - `id (uuid, pk)`
    - `email (text, unique, nullable)`
    - `phone (text, unique, nullable)`
    - `password_hash (text)`
    - `role (OWNER|TENANT)`
    - `status (ACTIVE|DISABLED)`
    - `created_at`, `updated_at`

**Constraints**

- Either `email` or `phone` must be present.
- Normalize `email` to lowercase; normalize `phone` to E.164.
- `refresh_tokens`
    - `id (uuid, pk)`
    - `user_id (fk users.id)`
    - `jti (text, unique)`
    - `expires_at (timestamptz)`
    - `revoked_at (timestamptz, nullable)`
    - `created_at`

#### S3.2 Property / room management

- `buildings`
    - `id (uuid, pk)`
    - `owner_id (fk users.id)`
    - `name (text)`
    - `address (text, nullable)`
    - `status (ACTIVE|ARCHIVED)`
    - `created_at`, `updated_at`
- `floors`
    - `id (uuid, pk)`
    - `building_id (fk buildings.id)`
    - `identifier (text)`
    - `status (ACTIVE|ARCHIVED)`

**Constraint**

- `price_rules`

#### S3.4 Contracts (1 active contract per room)

### S5. Key sequences / flows (MVP)

#### S5.1 Registration / Login / RBAC

1. Register/login returns tokens with role claim `OWNER` or `TENANT`.
2. API guard validates JWT.
3. Policy checks:
    - Owner endpoints require role Owner.
    - Tenant endpoints require role Tenant.
    - Resource checks ensure `owner_id` matches.

#### S5.2 Effective pricing (no overlaps)

1. Owner creates a new `price_rule` for (room, service) with `effective_start`.
2. Service checks latest rule.
3. If overlap would occur → reject `409 OVERLAPPING_EFFECTIVE_PERIOD`.
4. Otherwise, set previous rule `effective_end = new_start - 1 day`.

#### S5.3 Capture → OCR → validate vs previous → finalize

1. `POST /readings/capture`: store image; create reading (DRAFT); enqueue OCR job.
2. OCR service returns `{value, confidence}`.
3. API validates numeric:
    - non-numeric → NEEDS_REVIEW
    - numeric → RECOGNIZED
4. Compare with previous finalized reading:
    - if decreased → NEEDS_REVIEW
    - else allow finalize
5. Owner manually corrects (if needed) and finalizes.

#### S5.4 Generate invoices (idempotent)

1. `POST /invoices/generate?period=` loads active contracts.
2. For each contract:
    - if invoice exists → skip / return conflict
    - else create invoice
3. Add lines:
    - RENT (fixed)
    - ELECTRIC/WATER (delta from readings)
    - SERVICES (from configured price rules)
4. If missing reading or negative consumption → set invoice `PENDING_CONFIRMATION`.
5. Else set invoice `UNPAID`.

### S6. Security, privacy, and audit (MVP)

- **Password hashing:** Argon2id.
- **Transport:** TLS 1.2+.
- **Signed URLs:** photo/document links must expire ≤ 15 minutes.
- **Tenant isolation:** tenant invoice endpoints verify ownership by contract_tenants.
- **Audit events:** login, reading override/correction, invoice status changes, contract create/end.

---

### S7. Error handling and standard response

**Standard error format**

- `{ code, message, details? }`

**Common error codes (MVP)**

- `401 UNAUTHORIZED`
- `403 FORBIDDEN`
- `409 ROOM_ALREADY_HAS_ACTIVE_CONTRACT`
- `409 OVERLAPPING_EFFECTIVE_PERIOD`
- `409 INVOICE_ALREADY_EXISTS`
- `409 MISSING_PRICE_CONFIGURATION`
- `400 INVALID_UNIT_PRICE`
